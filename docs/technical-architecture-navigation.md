# 慧视 App 出行模式（导航 + 障碍物识别并行）技术方案文档

> 适用范围：首次开发阶段、团队协作对接、后端与小程序联调、后续功能扩展。
>
> 当前决策：采用“模块解耦 + 状态共享 + WebSocket 实时同步”的方案，出行模式下支持实时引导；障碍物识别结果仅用于提醒用户，不直接改写导航路径。

---

## 1. 文档目标

本文件用于统一项目中“出行模式”的研发思路，解决以下问题：

1. 导航功能与障碍物识别功能需要同时运行，互不阻塞。
2. 小程序前端、Python 后端、地图能力、高德服务之间的职责边界要清晰。
3. 团队第一次做应用开发，需要有统一、可交接、可拆分的技术路线。
4. 后续功能还会继续增加，因此架构必须便于扩展，而不是只满足当前 demo。

---

## 2. 项目背景与当前状态

### 2.1 项目定位

慧视 App 面向视障用户，核心目标是提供两大能力：

- **出行模式**：实时导航 + 障碍物提醒 + 场景引导
- **会视模式**：场景识别、物品讲解、文字阅读等能力

本技术方案只聚焦于 **出行模式**，因为当前任务的重点是：

- 完成导航功能
- 与障碍物识别同时运行
- 采用高德地图
- 方便与后端 Python 项目对接

### 2.2 当前已具备的基础

根据当前代码状态，小程序已经有以下内容：

- `pages/index/index`：主页 demo
- `pages/navigation/navigation`：导航页 demo
- `app.js`：有一个简易的障碍物监测模拟逻辑
- `app.json`：已经注册导航页面

当前 demo 的特点：

- 页面简陋，但能跑通基本流程
- 导航逻辑只是演示，没有真正接地图与路径能力
- 障碍物识别尚未接入真实后端
- 尚未建立前后端统一协议

### 2.3 当前阶段的核心问题

1. 导航和障碍物识别如何并行运行？
2. 谁负责导航逻辑，谁负责识别逻辑？
3. 前端如何拿到识别结果和导航状态？
4. 后端如何设计成可扩展结构，方便以后其他人接手？

## 3. 总体技术路线

### 3.1 已确认的方案

本项目采用以下决策：

- **方案 B：模块解耦 + 状态共享**
- **实时同步方式：WebSocket**
- **导航方式：实时引导**
- **障碍物识别影响导航的方式：仅提醒用户，不自动改道**
- **地图服务：高德地图**

### 3.2 总体架构图

```text
┌─────────────────────────────┐
│          小程序前端          │
│  页面展示 / 用户交互 / 状态显示 │
└──────────────┬──────────────┘
               │ HTTP + WebSocket
               ▼
┌─────────────────────────────┐
│          Python 后端         │
│  API 层 / 状态层 / 推送层       │
└───────┬─────────────┬───────┘
        │             │
        ▼             ▼
┌─────────────┐   ┌─────────────┐
│ 导航服务模块  │   │ 识别服务模块  │
│ 路线/状态/引导 │   │ 摄像头/模型/提醒 │
└──────┬──────┘   └──────┬──────┘
       │                  │
       └──────共享状态中心──┘
                │
                ▼
         WebSocket 实时推送
```

### 3.3 设计原则

1. **职责分离**：每个模块只做自己的事情。
2. **状态共享**：导航状态和识别状态统一维护。
3. **实时同步**：前端不依赖频繁轮询，优先使用 WebSocket。
4. **可扩展**：后续加入语音、室内导航、报警、历史记录时，不推翻现有结构。
5. **便于协作**：接口、字段、状态名称统一，减少沟通成本。

## 4. 功能范围定义

### 4.1 出行模式功能范围

出行模式只包含与导航相关的能力，不包括会视模式的详细讲解或阅读文字功能。

#### 必须具备

- 目的地输入
- 路线规划
- 开始导航 / 停止导航
- 实时导航引导
- 障碍物提醒
- WebSocket 状态同步
- 高德地图能力接入

#### 暂不做或后做

- 自动改道
- 复杂室内导航
- 多终端协同导航
- 历史路线记录
- 高级个性化策略

### 4.2 导航的定义

本项目中的“导航”不是简单显示地图，而是面向视障用户的实时引导，包括：

- 前方路口左转 / 右转
- 前方直行
- 距离目标地点还有多少米
- 当前是否偏航
- 前方是否存在障碍物提醒

### 4.3 障碍物识别的定义

障碍物识别模块负责检测用户前方环境中的风险，例如：

- 台阶
- 行人
- 自行车
- 路障
- 车辆
- 桌椅
- 坑洼

识别结果的作用：

- 产生提醒
- 播报风险级别
- 必要时提示减速或停止前进

但不直接改变导航路线。

## 5. 模块划分与职责边界

### 5.1 小程序前端模块

#### 负责内容

- 页面结构和交互
- 导航页与首页切换
- 目的地输入
- 开始/停止导航按钮
- 接收并展示后端状态
- WebSocket 长连接管理
- 障碍物提醒与导航播报内容展示

#### 不负责内容

- 视觉识别计算
- 地图路径推理核心逻辑
- 复杂的导航状态调度
- 任务并发调度

### 5.2 Python 后端模块

#### 负责内容

- 接收前端请求
- 管理导航状态
- 管理障碍物识别状态
- 调用高德开放平台接口
- 向前端推送实时状态
- 管理识别服务与导航服务的并行运行

#### 不负责内容

- 前端 UI 展示
- 小程序页面跳转
- 复杂交互动画

### 5.3 导航服务模块

导航服务模块建议独立出来，职责是：

- 接收目的地
- 获取当前位置
- 规划路线
- 维护导航会话
- 生成实时引导文本
- 处理偏航状态

### 5.4 障碍物识别模块

职责是：

- 采集摄像头帧或图像
- 调用模型进行检测
- 生成障碍物提醒
- 输出风险等级、类型、方向、距离等信息

### 5.5 状态中心

状态中心是整个方案的核心，不是数据库，而是一个统一的数据层，保存：

- 导航状态
- 障碍物状态
- 设备状态
- 播报状态
- 当前会话信息

它的作用是：

- 让导航和识别模块共享状态
- 让 WebSocket 可以直接推送最新状态
- 让前端随时看到一致的数据

## 6. 并行运行设计

### 6.1 为什么不能把所有逻辑放在一个流程里

如果导航和识别都写在同一条同步执行链上，会出现以下问题：

- 一个模块阻塞另一个模块
- 摄像头识别慢时，导航响应变差
- 导航更新频率高时，识别结果会丢失
- 代码越来越难维护

因此必须拆成独立运行的模块。

### 6.2 推荐并行方式

结合当前项目阶段，推荐如下：

#### Python 后端层

- 导航服务独立运行
- 识别服务独立运行
- 共享状态中心统一存储状态
- WebSocket 负责实时推送给前端

#### 并发模型建议

如果识别任务是模型推理密集型：

- 优先使用 **多进程**

如果识别任务只是采集、转发、轻量处理：

- 可使用 **线程 + 队列**

如果后续实时性要求更高：

- 采用 **异步框架 + WebSocket**

### 6.3 当前阶段推荐落地方式

为了首次开发不复杂，建议：

- 后端主进程提供 API 和 WebSocket
- 识别任务作为独立 worker 运行
- 导航任务作为独立 worker 运行
- 两个 worker 通过状态中心交换信息

这样可以保证：

- 互不阻塞
- 方便调试
- 方便团队分工

## 7. WebSocket 状态同步方案

### 7.1 为什么选择 WebSocket

因为本项目是实时引导场景，前端需要持续获取：

- 当前导航状态
- 当前障碍物提醒
- 播报内容
- 设备状态变化

如果使用纯 HTTP 轮询：

- 实时性一般
- 请求次数多
- 状态变化不够平滑

WebSocket 更适合：

- 状态连续推送
- 低延迟更新
- 语音播报联动
- 后续扩展提醒机制

### 7.2 WebSocket 推送内容

建议推送以下类型的数据：

- `navigation_status`
- `obstacle_alert`
- `speech_update`
- `device_status`
- `session_state`

### 7.3 前端处理方式

前端维护一个 WebSocket 连接：

- 连接成功后订阅状态
- 后端状态变化时主动推送
- 前端收到后更新页面显示和语音播报区域
- 若连接断开，自动重连

### 7.4 断线重连原则

由于移动端网络不稳定，必须考虑：

- 自动重连
- 重连退避策略
- 断线后显示“连接中”状态
- 恢复后同步最新状态

## 8. 高德地图接入思路

### 8.1 高德地图的作用

高德负责提供：

- 地图展示
- 地点搜索
- 路线规划
- 当前位置定位
- 导航基础能力

### 8.2 本项目对高德的使用策略

建议分成两层：

#### 第一层：地图能力层

- 直接使用高德开放平台能力
- 获取地理位置和路线数据
- 展示基础路线信息

#### 第二层：业务引导层

- 由本项目自己生成适合视障用户的语音引导
- 结合障碍物识别结果进行提醒
- 生成更口语化、更贴近用户的指令

### 8.3 为什么不能完全依赖地图默认能力

因为普通地图导航主要面向视觉用户，视障场景还需要：

- 更高频的文字播报
- 更明确的危险提示
- 更短更易理解的引导语
- 与障碍物提醒联动

所以高德提供底层能力，本项目提供无障碍增强层。

## 9. 数据与状态设计

### 9.1 统一状态结构建议

建议定义统一状态对象，包含：

- `navigation`
  - `isNavigating`：是否正在导航
  - `destination`：目的地
  - `routeSummary`：路线摘要
  - `directionHint`：当前引导语
  - `isOffRoute`：是否偏航
  - `eta`：预计到达时间

- `obstacle`
  - `hasObstacle`：是否有障碍物
  - `type`：障碍物类型
  - `distance`：距离
  - `riskLevel`：风险等级
  - `hint`：提醒内容

- `device`
  - `cameraStatus`：摄像头状态
  - `gpsStatus`：定位状态
  - `networkStatus`：网络状态

- `speech`
  - `isSpeaking`：是否播报中
  - `content`：当前播报内容
  - `mode`：播报类型

### 9.2 状态设计原则

- 字段命名统一
- 尽量短小清晰
- 可扩展
- 前后端同名

### 9.3 为什么要统一状态结构

这样前端、后端、算法同学可以对着同一份字段工作，避免：

- A 同学叫“危险等级”，B 同学叫“风险级别”
- A 同学返回数组，B 同学返回字符串
- 前端接收字段时频繁改代码

## 10. 接口设计建议

以下接口是建议版，后续可根据后端框架微调。

### 10.1 获取当前状态

**接口**：`GET /api/status`

**用途**：
- 小程序进入页面时拉取初始状态
- WebSocket 重连后补状态
- 页面刷新后恢复当前会话

### 10.2 开始导航

**接口**：`POST /api/navigation/start`

**请求参数**：
- `destination`
- `latitude`
- `longitude`
- `mode`

**作用**：
- 创建导航会话
- 规划路线
- 启动导航 worker
- 开始实时引导

### 10.3 停止导航

**接口**：`POST /api/navigation/stop`

**作用**：
- 停止当前导航会话
- 清理状态
- 停止相关播报

### 10.4 获取路线信息

**接口**：`GET /api/navigation/route`

**参数**：
- `destination`

**作用**：
- 返回路线概要
- 支持路线预览

### 10.5 上报障碍物识别结果

**接口**：`POST /api/obstacle/report`

**作用**：
- 识别模块上报检测结果
- 后端统一决定是否提醒用户
- 更新状态中心

### 10.6 WebSocket 状态推送

**接口**：`WS /ws/status`

**作用**：
- 实时推送导航与识别状态
- 保持前端页面同步

## 11. 障碍物识别如何影响导航

### 11.1 当前决策

本项目采用：

- **只提醒用户**
- **不自动改道**
- **不直接中断导航流程，除非风险很高**

### 11.2 为什么先这样设计

因为自动改道涉及：

- 路径重算
- 位置误差控制
- 场景复杂度高
- 误判风险更大

第一次开发时建议先保证稳定性与可解释性，因此先做“提醒”而不是“自动干预路径”。

### 11.3 提醒方式建议

障碍物提醒可以包含：

- 障碍物类型
- 距离
- 方向
- 风险等级
- 建议动作

示例：

- “前方 2 米有台阶，请减速”
- “左前方有行人，请注意避让”
- “前方有自行车，请暂停前进”

## 13. 后端 Python 项目建议结构

建议后端目录按职责拆分：

```text
PythonProject/
├── app/
│   ├── api/
│   ├── services/
│   ├── workers/
│   ├── models/
│   ├── utils/
│   └── config/
├── tests/
├── requirements.txt
└── README.md
```

### 13.1 模块说明

- `api/`：对外提供 HTTP 和 WebSocket 接口
- `services/`：业务逻辑层，例如导航服务、识别服务
- `workers/`：持续运行任务，例如摄像头检测、状态上报
- `models/`：数据结构和状态定义
- `utils/`：日志、配置、通用函数
- `config/`：环境变量、密钥、地图配置

### 13.2 为什么要这样拆

这样后续可以让不同同学分别接：

- 接口同学做 `api/`
- 算法同学做 `workers/`
- 业务同学做 `services/`
- 联调同学看统一状态文档即可

## 17. 风险与应对

### 风险 1：导航与识别互相阻塞

**应对**：模块解耦，后台 worker 独立运行，状态通过共享层同步。

### 风险 2：接口字段混乱

**应对**：统一状态协议，提前写文档并固定字段名。

### 风险 3：WebSocket 不稳定

**应对**：加自动重连和状态补拉。

### 风险 4：高德能力接入成本高

**应对**：先实现最小可用版本，再逐步扩展。

### 风险 5：第一次开发经验不足

**应对**：按阶段推进，先跑通链路，再逐步优化体验。

### 里程碑 B：出行模式完成

### 里程碑 C：实时引导完成

- 导航播报与识别提醒同时生效
- WebSocket 实时推送稳定
- 页面交互适合视障用户

## 20. 结论

本项目的出行模式建议采用以下最终方案：

- **导航是实时引导型导航**
- **障碍物识别只做提醒，不直接改道**
- **导航和识别采用模块解耦、状态共享的方式运行**
- **前后端通过 WebSocket 做实时状态同步**
- **地图能力优先接入高德地图**
- **后端 Python 项目按服务化、模块化方式拆分，便于团队协作和后续扩展**

这份方案适合作为第一次开发应用的基础交接文档，后续只需要在此基础上逐步补齐接口、状态和 UI 即可。

## 21. 附录：建议统一术语

- **出行模式**：导航、路线、提醒相关功能
- **会视模式**：场景识别、文字读取、讲解相关功能
- **实时引导**：导航过程中的持续语音提示
- **障碍物提醒**：识别到风险后提示用户注意
- **状态中心**：统一管理当前导航/识别/设备状态的数据层
- **worker**：持续执行某项任务的后台进程或线程
- **WebSocket**：前后端实时通信协议
